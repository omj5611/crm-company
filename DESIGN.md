# 통합관리솔루션

## 1. 기본 원칙

- 정보 확인이 가장 우선이다.
- 장식보다 구조와 정렬을 먼저 잡는다.
- 화면은 흰색 배경을 기본으로 한다.
- 회색 계열의 선과 면으로 구획을 나눈다.
- 과한 그림자, 강한 라운드, 불필요한 이미지 요소는 사용하지 않는다.
- 카드 UI는 필요한 곳에만 최소한으로 사용한다.
- 텍스트는 `Pretendard`만 사용한다.
- 폰트 굵기는 `400`, `500`, `600`까지만 사용한다.

## 2. 톤과 밀도

- 전체 분위기는 실무형, 정돈됨, 절제된 느낌을 유지한다.
- 기본 폰트 크기는 작고 단정하게 유지한다.
- 핵심 제목만 상대적으로 크게 보여준다.
- 본문, 라벨, 보조 정보는 비슷한 밀도로 묶는다.
- 의미 없는 강조는 피하고, 필요한 부분만 블루로 드러낸다.

## 3. 레이아웃 규칙

- 전체 페이지는 넓은 흰색 캔버스처럼 보이게 한다.
- 섹션 간 간격은 일정하고 충분히 여백을 둔다.
- 패딩과 마진은 기능별 리듬을 먼저 맞춘다.
- 같은 역할의 영역은 같은 간격 규칙을 따른다.
- 타이틀과 본문은 같은 카드 안에서 너무 분리하지 않는다.
- 타이틀 영역은 불필요하게 카드처럼 두껍게 감싸지 않는다.

## 4. 타이포그래피

### Font Family

```css
font-family: "Pretendard";
```

### 권장 크기

- 페이지 타이틀: `1.2rem` 이상
- 섹션 타이틀: `1rem` 내외
- 본문: `0.8rem ~ 0.88rem`
- 메타 정보: `0.76rem ~ 0.8rem`

### 권장 굵기

- `400`: 본문, 설명, 보조 문장
- `500`: 라벨, 상태, 칩, 보조 강조
- `600`: 타이틀, 핵심 수치, 현재 선택값

### 금지

- `700` 이상 사용 금지
- 장식성 서체 사용 금지
- 불필요한 대문자 강조 금지

## 5. 디자인시스템 색상 토큰

- `color.surface.raised = #f9fafb`
- `color.surface.strong = #e5e7eb`
- `color.surface.overlay = rgba(17, 24, 39, 0.45)`

### Text

- `color.text.secondary = #374151`
- `color.text.tertiary = #1f2937`
- `color.text.inverse = #9ca3af`
- `color.text.placeholder = #9ca3af`

### Border and line

- `color.border.subtle = #e5e7eb`
- `color.border.default = #d1d5db`
- `color.border.strong = #9ca3af`

### Blue accent

- `color.accent.blue.100 = #dbeafe`
- `color.accent.blue.200 = #bfdbfe`

### Neutral and state colors

### Usage rules

- 컴포넌트는 가능하면 원시 색상보다 의미 기반 토큰을 사용해야 한다.
- 기본 배경은 `color.surface.base` 또는 `color.surface.muted`를 사용해야 한다.
- 입력, 테이블, 카드 경계는 `color.border.subtle` 또는 `color.border.default`를 사용해야 한다.
- 강조 상태는 `color.accent.blue.*` 범위 내에서만 처리해야 한다.
- 경고와 오류는 `color.state.warning`, `color.state.danger`만 사용해야 한다.
- 그림자보다 경계선과 간격을 우선해야 한다.

## 6. 형태 규칙

- 기본 라운드는 작고 얇게 유지한다.
- 모서리는 둥글지만 두껍게 보이지 않게 한다.
- 버튼, 입력, 셀렉트, 칩은 통일된 높이를 사용한다.
- 표와 목록은 과한 카드감보다 정보 정렬을 우선한다.
- 구분선은 얇고 낮은 대비로 처리한다.

## 7. 시각적 표현

- 그림자는 기본적으로 사용하지 않는다.
- 꼭 필요한 경우에도 아주 약한 경계선 정도만 사용한다.
- 입체감은 그림자보다 간격과 선으로 표현한다.
- 상태는 색보다 레이블과 구조를 먼저 읽히게 한다.
- 배경 장식, 패턴, 과한 그래픽 요소는 사용하지 않는다.

## 8. 컴포넌트 규칙

### 공통 원칙

- 같은 역할의 UI는 하나의 공통 컴포넌트를 재사용해야 한다.
- 내용만 다른 요소는 동일한 렌더 구조와 동일한 스타일을 공유해야 한다.
- 필드, 버튼, 메뉴, 툴팁, 다이얼로그는 각각 공통 컴포넌트 계층을 둬야 한다.
- 같은 기능이면 화면마다 다른 모양을 만들지 않아야 한다.
- 같은 기능의 다이얼로그는 `company-tag-dialog__panel` 같은 공통 패널 구조를 재사용해야 한다.
- 동일한 동작을 하는 다이얼로그는 디자인과 인터랙션을 통일해야 하며, 저장 위치와 반영 대상만 화면별로 다르게 연결해야 한다.

### 버튼

- 기본 버튼과 보조 버튼은 같은 베이스 스타일을 사용해야 한다.
- 주요 동작은 블루 계열로만 강조해야 한다.
- 보조 동작은 회색 기반으로 유지해야 한다.
- 버튼은 높이와 패딩을 통일해야 한다.

### 입력 필드

- 검색, 일반 입력, 셀렉트, 텍스트영역은 같은 필드 규칙을 따라야 한다.
- 라벨은 작고 명확하게 둬야 한다.
- 입력 컨트롤은 너비를 100%로 맞춰야 한다.
- 필드 간격은 좁고 일정하게 유지해야 한다.

### 필터

- 필터는 개별 UI가 아니라 공통 필드 컴포넌트로 구성해야 한다.
- 선택형 필터는 같은 셀렉트 스타일을 재사용해야 한다.
- 검색 필드는 같은 입력 스타일을 재사용해야 한다.

### 표

- 표는 목록 확인에 최적화된 구조로 유지해야 한다.
- 헤더와 본문은 얇은 선으로만 구분해야 한다.
- 정보 밀도가 높아도 읽기 쉬운 간격을 유지해야 한다.
- 상태값은 칩 형태로 가볍게 표현해야 한다.

### 다이얼로그

- 다이얼로그는 별도 top-layer 성격으로 다뤄야 한다.
- 본문 위 다른 섹션에 가려지지 않아야 한다.
- 등록, 편집, 확인 같은 목적별로 동일한 카드 구조를 사용해야 한다.
- 다이얼로그 내부 필드도 공통 필드 컴포넌트를 재사용해야 한다.
- 같은 유형의 다이얼로그는 공통 제목, 본문, 버튼 배치, 닫기 동작, 포커스 규칙을 공유해야 한다.
- 데이터 저장 대상만 다르면 동일한 다이얼로그 패널을 재사용하고, 저장 시 반영되는 테이블이나 필드만 분기해야 한다.

### 메뉴 / 드롭다운

- 메뉴와 드롭다운은 공통 오버레이 컴포넌트로 만들어야 한다.
- 메뉴는 섹션 내부가 아니라 독립 레이어에서 렌더링해야 한다.
- 바깥 클릭, `Esc`로 닫을 수 있어야 한다.
- 배치가 필요하면 뷰포트 기준으로 계산해야 한다.

### 툴팁

- 툴팁은 공통 오버레이 컴포넌트를 사용해야 한다.
- 항상 아이콘 기준 상단에 떠야 한다.
- 검정 배경을 사용하고, 말꼬리 삼각형은 사용하지 않아야 한다.
- 섹션, 카드, 표, 스크롤 영역에 의해 가려지면 안 된다.
- 툴팁 위치는 화면 가장자리를 고려해 자동 보정해야 한다.

## 9. 오버레이 규칙

- 툴팁, 메뉴, 드롭다운, 다이얼로그, 확인 모달은 모두 공통 오버레이 정책을 따라야 한다.
- 오버레이는 가능한 한 최상위 레이어에서 렌더링해야 한다.
- 섹션 내부 `overflow`에 영향을 받지 않도록 분리해야 한다.
- 활성 오버레이는 항상 다른 내용보다 위에 있어야 한다.
- 스크롤과 리사이즈 시 위치를 다시 계산해야 한다.
- 메모 툴팁, 모달, 드롭다운 메뉴는 항상 뷰포트 안에서 렌더링해야 하며 창 밖으로 잘리면 안 된다.
- 오버레이가 가장자리에 닿을 수 있으면 화면 안쪽으로 자동 재배치하거나 고정 포털 레이어로 렌더링해야 한다.
- 오버레이 내부 내용이 길어질 경우에는 외부를 넘기지 말고 내부 스크롤이나 최대 높이 제한으로 처리해야 한다.

## 10. 확인 모달 규칙

- 삭제, 닫기, 초기화 같은 되돌리기 어려운 동작에는 항상 확인 모달을 띄워야 한다.
- 확인 모달은 공통 컴포넌트로 재사용해야 한다.
- 문구만 바꿔서 같은 모달 구조를 사용해야 한다.
- 사용자가 명시적으로 확인해야만 동작이 실행돼야 한다.

## 11. 상태 표현

- 상태는 칩과 라벨로 표현해야 한다.
- 정상, 주의, 경고, 정보 상태는 시각적으로 구분해야 한다.
- 상태 색은 과하게 진하지 않게 사용해야 한다.
- 의미가 있는 경우에만 회색을 적극 활용해야 한다.

## 12. 접근성 요구사항 및 테스트 기준

- 모든 상호작용 요소는 `focus-visible` 상태를 보여줘야 한다.
- 모든 상호작용 요소는 포인터와 동일한 키보드 경로를 가져야 한다.
- 모든 폼 필드는 연결된 라벨을 가져야 한다.
- 의미가 있는 상태 변화는 텍스트 또는 접근 가능한 이름 변화로 전달돼야 한다.
- 텍스트와 컨트롤의 대비는 항상 충분해야 한다.
- 키보드만으로 버튼, 링크, 입력, 탭, 메뉴, 드롭다운, 다이얼로그를 순서대로 도달하고 조작할 수 있어야 한다.
- 포커스는 오버레이나 클리핑 영역 뒤로 사라지면 안 된다.
- 선택 상태, 현재 페이지, 현재 탭은 프로그램적으로 식별 가능해야 한다.
- 에러 메시지는 실패한 필드와 연결되어야 한다.
- 로딩 상태 컨트롤은 중복 제출이 불가능해야 한다.
- 200% 확대에서도 주요 흐름의 콘텐츠가 사라지지 않아야 한다.

## 13. 콘텐츠와 톤 표준

- 문구는 간결해야 한다.
- 문구는 자신감 있고 구현 중심이어야 한다.
- 라벨은 대상이나 동작을 명확히 설명해야 한다.
- 버튼 텍스트는 행동의 결과를 말해야 한다.
- 도움말은 라벨을 반복하지 말고 다음 행동을 알려야 한다.
- 빈 상태는 빠진 정보와 다음 단계를 설명해야 한다.

### 예시

- 좋은 예: `사업 관리`
- 좋은 예: `지원자 관리`
- 좋은 예: `정산 증빙 파일 업로드`
- 좋은 예: `검색 결과가 없습니다. 검색어를 바꾸거나 필터를 초기화해 주세요.`
- 피해야 할 예: `확인`, `보기`, `처리`, `클릭`
- 피해야 할 예: `기능 준비 중입니다`

### 제외 규칙

- 화면 본문에는 개발 설명 문구를 넣지 않아야 한다.
- `MEMBER MANAGEMENT` 같은 부가 메타 텍스트는 노출하지 않아야 한다.
- 페이지 설명은 기능 요약이 아니라 사용자가 바로 이해할 수 있는 핵심 제목 중심으로 구성해야 한다.

## 14. 금지 요소

- 과한 카드 중첩
- 두꺼운 그림자
- 강한 라운드 박스
- 불필요한 이미지
- 장식용 아이콘 남용
- 굵은 타이포 계층
- 서로 다른 페이지에서 같은 기능을 다른 UI로 구현하는 것

## 15. 적용 대상

- 대시보드
- 회원 목록
- 교육과정 등록 다이얼로그
- 필터와 검색 영역
- 툴팁과 도움말
- 드롭다운 메뉴
- 확인 모달

## 16. QA 체크리스트

- 기본 배경이 흰색으로 유지되는가
- 회색 계열 선과 면으로 영역이 구분되는가
- `Pretendard`만 사용되는가
- `400`, `500`, `600`만 사용되는가
- 버튼, 입력, 셀렉트, 칩 높이가 일관적인가
- 카드가 꼭 필요한 곳에만 사용되는가
- 그림자 없이도 구조가 읽히는가
- 모든 인터랙션에 `focus-visible`가 보이는가
- 오버레이가 스크롤 영역에 가려지지 않는가
- 에러 상태가 색과 텍스트로 모두 전달되는가
- 긴 라벨과 긴 내용이 레이아웃을 깨지 않는가
- 빈 상태가 다음 행동을 안내하는가
- 키보드만으로 주요 작업을 끝낼 수 있는가

## QA checklist
- Confirm all interactive elements have visible `focus-visible` styling.
- Confirm keyboard navigation reaches every interactive control in the intended order.
- Confirm hover, active, disabled, loading, and error states are implemented for every component.
- Confirm long labels, long values, and empty states do not break layout.
- Confirm text contrast passes WCAG 2.2 AA in default and elevated surfaces.
- Confirm mobile layout stacks without hidden content or broken actions.
- Confirm dialogs, menus, and tabs are operable without a mouse.
- Confirm semantic tokens are used in component guidance and implementation.
- Confirm no one-off spacing or typography values were introduced without review.
- Confirm any loading action cannot be submitted twice.
